# Explore Planning Rules

> 適用範圍：規劃或重構 `/explore` 主題導覽、`src/content/explore/*.md`、以及 `src/components/explore/*ExploreRoot.tsx` 的 guide / compare 模式。  
> 視覺規格見 [`exploreart.md`](exploreart.md)，文案結構見 [`textstyle.md`](textstyle.md)，工程邊界見 [`p5toreact.md`](p5toreact.md)。

---

## 1. 目的與邊界

Explore 是主題導覽與概念入口，負責建立多件 Works 之間的學習路線、共同語言、角色切換或限制條件；不是 Works 摘要清單，也不是把多件 Work 合併成一件。

| Explore 要做 | Explore 不做 |
|--------------|--------------|
| 把多件 Works 串成一條中心問題與學習路線 | 逐條重講每件 Work 的核心結論 |
| 說明同一個概念在不同表徵中的角色 | 把 Works 的公式、動畫和觀察重點拼接 |
| 幫讀者建立讀圖語言 | 取代 Works 的單件技術細節 |
| 降低預設導覽模式的控制量 | 在預設畫面堆疊所有作品的完整控制 |

### Works 與 Explore 的關係

Explore 是主題導覽，Works 是單件實驗。`/works` 必須能獨立呈現單一數學物件、公式或幾何現象；`/explore` 負責建立概念地圖、學習路線與相關 Works 的串接。兩者是「導覽 — 實驗」配套，不是合併，也不是互相複製。

| 區域 | 主要結構 | 邊界 |
|------|----------|------|
| Works | `.work-detail__stage` + `.work-detail__canvas` + `.controls-panel--stage` | 使用 `CurveModule`、work registries、`WorkInteractiveStage` 與 portal 控制面板 |
| Explore | `*-explore__stage` + `*-explore__visual` + `*-explore__sidebar` suffix contract | 使用 `src/explore/*`、explore registries 與內嵌 sidebar，不走 Works portal / `CurveModule` |

內容重疊時，Explore 保留主題背景、概念關係、學習順序與導覽；Work 保留單一互動、公式細節、參數定義與觀察任務。最簡判斷：刪掉 Work 時，Explore 仍能說明主題地圖；刪掉 Explore 時，Work 仍是一個完整作品。

規劃時先問一句：

```text
這個 Explore 要回答「同一個什麼」如何在多個情境中反覆出現、改變角色，或被限制？
```

如果回答只是「A 作品講 X，B 作品講 Y，C 作品講 Z」，代表還停在清單層級。

## 2. 規劃順序

1. 列出相關 Works，寫下每件 Work 的單件核心，不超過一句。
2. 寫出 Explore 的中心句：同一個概念、同一個公式、同一個操作、或同一種讀圖語言。
3. 標出 Explore 必須避免重複的 Works 結論。
4. 判斷改動層級：content-only、light UX、或 guide-mode refactor。
5. 定義預設模式：畫面分區、顯示控制、隱藏控制、手機排版。
6. 定義可測試 helper、邊界行為、browser 驗證項目。

不要先設計畫面再找主題。主題句不清楚時，畫面通常會變成多件作品的同屏堆疊。

---

## 3. 去重規則

- `觀察重點` 不能和 `相關作品` 一對一對應。
- 若某條 observation 可以直接當成某件 Work 的標題或核心句，必須改寫成跨作品關係。
- 可以在 `相關作品` 列清單；不要在 `觀察重點` 寫清單式摘要。
- 不重複單件 Work 的公式，除非是為了說明同一公式在另一個表徵中改變角色。
- 對照模式只能共用語意相同的控制參數；若參數物理或數學意義不同，必須分開控制或明確標成類比。
- 類比不能假裝成同一機制。若某面板只是概念類比，UI 和文案都要明說。

常見改寫方向：

| 重複型寫法 | 導覽型寫法 |
|------------|------------|
| `n 增大時黎曼和更接近積分；割線趨近切線。` | `同一個尺度縮小動作，分別把全域累積誤差與局部曲率殘差壓小。` |
| `相位差決定相長或相消。` | `相位關係在不同媒介中轉成位移疊加、條紋平移或節點位置。` |
| `內積表示投影，法向量表示垂直。` | `向量在位置、方向、座標三種角色間切換，形成讀圖語言。` |
| `帕斯卡三角形、路徑計數、分支爆炸。` | `同一個 C(n,k) 在係數、路徑和遞迴依賴中反覆出現。` |

## 4. Guide Mode 規則

預設 guide / compare 模式應該減少認知負荷，不是把所有 Works 的完整動畫放在一起。

- 預設模式只保留能支撐中心句的控制。
- 若三個面板同時可見，焦點切換只能改變高亮與統計文字；手機版可改成垂直堆疊或縮略呈現，但必須在計劃中明說。
- 若某面板需要額外輔助物件，例如投影用的第二向量 `u`，要定義初始值、可拖動範圍，以及是否只在該面板顯示。
- 若基底、參考點、取樣點或相位來源是固定的，要寫進 interface / assumptions，不讓實作者猜。
- 若模式切換會保留或重置狀態，要明確定義。
- 統計區要有數值，但不能塞滿 Works 級技術說明。

共享控制的判斷：

| 可以共享 | 不應直接共享 |
|----------|--------------|
| 同一尺度同時控制分割細度與局部跨度，但兩者映射各自獨立 | 把 `h = Δx` 當成同一個量 |
| 同一相位差推動兩列波的相對初相 | 用相位差直接驅動 Chladni 本徵節線 |
| 同一點 `p` 在不同面板改變角色 | 讓所有輔助向量出現在所有面板造成干擾 |

## 5. Content 規則

`src/content/explore/*.md` 仍遵守 [`textstyle.md`](textstyle.md) 的固定五段結構。規劃時先寫每段功能：

| 區塊 | 規劃要求 |
|------|----------|
| `description` | 寫主題關係，不寫 Works 清單 |
| `基本概念` | 寫共享機制與角色差異 |
| `互動說明` | 寫讀者操作什麼、看見什麼 |
| `觀察重點` | 寫 3 條跨表徵結論，避免控制操作語句 |
| `相關作品` | 可以列 Works，但不把 Works 摘要搬到 observation |
| `延伸閱讀` | 補外部脈絡，不承擔核心導覽 |

需要區分「同一公式」與「限制後的子集合」。例如 Catalan number 不是另一個 `C(n,k)`，而是從 `C(2n,n)` 的候選集合中加入限制條件後留下的合法子集合。

## 6. Interfaces And Tests

計劃若涉及實作，必須把以下項目寫成可驗證條件：

- public types / mode names / default mode。
- 每個模式顯示哪些控制，隱藏哪些控制。
- 參數映射公式、上下界、clamp 行為。
- helper 的輸入輸出與邊界案例。
- sidebar stats 不得為空；每個 role / mode 至少有一條可測試數值。
- label helper 要測邊界，例如 `k = 0`、`k = n`、`phase = 0.5`、`h` 越界。
- browser 驗證要包含 desktop、mobile、模式切換、控制數量、動態互動、無 overflow。

測試描述要能變成斷言。避免只寫「文案不重複」、「畫面有意義」這類無法自動或半自動驗證的項目；必要時改成關鍵詞、非空陣列、邊界 label、DOM 控制數量或 screenshot 行為。

---

## 7. Review Checklist

- [ ] 中心句是否說清楚「同一個什麼」？
- [ ] 每條 observation 是否都跨越至少兩個表徵？
- [ ] 是否避免把 Works 的核心結論搬進 Explore？
- [ ] 對照面板的共享參數是否語意一致？
- [ ] 類比和同一機制是否被清楚區分？
- [ ] 預設 guide 模式是否比專注模式更簡潔？
- [ ] 額外輔助物件、固定點、固定基底是否定義清楚？
- [ ] 手機排版是否和「所有面板可見」等要求相容？
- [ ] stats / labels / helpers 是否有邊界測試？
- [ ] browser 驗證是否包含互動，而不只是靜態版面？
