# 數學內容審查 Checklist

Date: 2026-06-13

這份 checklist 用於 Lab 維護期的人工數學內容審查。它補足 `audit:content` 無法檢查的語意風險；不取代 frontmatter、cover、build、registry 或互動整合檢查，也不新增任何 release tooling。

## 使用時機

- 新增 Works 或 Explore 文章時使用。
- 修改已公開文章的數學敘述時使用。
- 社團發布、課堂分享、正式對外分享前使用。
- 若不確定數學敘述是否正確，標成待確認，不要硬公開。

## 1. 定義檢查

- 重要名詞是否定義清楚。
- 是否避免只靠直覺描述帶過核心概念。
- 是否說明定義的使用範圍。
- 同一個詞是否在全文維持同一個意思。
- 讀者是否能從正文知道符號、圖像、互動參數各代表什麼。

## 2. 例外條件

- 分母是否可能為 0。
- 參數為 0 或負值時，敘述是否仍成立。
- 邊界情況是否寫明，例如端點、極限、空集合、單一元素。
- 不連續、漸近線、退化情形是否有明確說明。
- 機率事件是否互斥 / 獨立；若不是，是否避免把它們當成互斥 / 獨立。
- 幾何物件是否可能重合、平行、退化成點或線。

## 3. 公式與參數

- 公式是否正確，且和互動圖、文字說明一致。
- 變數符號是否一致；不要在同一頁讓同一符號代表不同量。
- 參數範圍是否清楚，例如整數、正數、非零實數、角度或弧度。
- 是否有過度簡化，例如把只在特例成立的結論寫成一般結論。
- 是否清楚區分近似、直觀描述、定理、公式定義與完整證明。
- 單位、座標軸、尺度、方向是否和圖像一致。

## 4. 教學語氣

- 高中生是否容易誤解這段話。
- 是否需要補「此為直觀說明，不是完整證明」。
- 是否有常見錯誤提醒，例如「相關不等於因果」、「樣本比例不等於真實機率」。
- 互動圖是否真的支撐文字說明；若圖只能展示特例，文字不可寫成通則。
- 是否避免用看似精確但未定義的詞，例如「很快」、「接近」、「穩定」而沒有說明比較基準。
- 是否把工程限制和數學結論分開，例如畫面裁切、取樣密度、動畫速度不應被寫成數學性質。

## 5. 審查流程

1. 先確認文章類型：Works、Explore、草稿、已公開文章。
2. 讀完正文，不先看互動圖；標出定義、公式、參數與結論。
3. 對照互動圖，確認圖像支撐文字，而不是暗示正文沒有說清楚的結論。
4. 檢查例外條件與邊界情況。
5. 以高中生視角重讀一次，標出可能誤解的句子。
6. 若有不確定的數學敘述，改成待確認或降級成直觀說明。
7. 修改後再跑既有 release gate，不新增工具。

## 6. 審查紀錄格式

```markdown
## Math Content Review Record

- date:
- page / slug:
- reviewer:
- checked items:
  - 定義:
  - 例外條件:
  - 公式與參數:
  - 教學語氣:
  - 互動圖支撐:
- concerns:
  - ...
- follow-up:
  - ...
```

## 判定原則

- `pass`：定義、例外條件、公式、參數與教學語氣都已人工確認。
- `concern`：敘述可能正確，但讀者容易誤解，需要補註或改寫。
- `blocked`：數學正確性不確定，需要查證或請人審閱；不要公開。
- `follow-up`：不阻塞本次修正，但應建立後續內容或互動改善項目。

## 明確非目標

- 不新增 script。
- 不修改 content。
- 不修改 registry。
- 不修改互動模組。
- 不擴張 Lab release system。
- 不把人工數學審查塞進 `audit:content`。
